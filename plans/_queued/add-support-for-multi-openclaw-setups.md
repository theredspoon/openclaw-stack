# Multi-OpenClaw Instance Architecture

## Context

Currently the VPS runs a single OpenClaw gateway container. We want to run multiple independent OpenClaw instances on the same VPS, each with its own configuration, agents, and permissions. This enables different "claws" for different use cases (personal, work, experimental) while sharing infrastructure (same VPS, same CF tunnel, same `openclaw:local` image, same sandbox images).

**WebSocket constraint**: The openclaw frontend hardcodes WebSocket connections to `wss://<host>/` (root path only, ignoring `controlUi.basePath`). This makes subpath routing on a single subdomain impossible without patching upstream. **Decision**: Use separate subdomains per instance (e.g., `personal.openclaw.domain.com`, `work.openclaw.domain.com`), all routed through the same CF tunnel.

---

## Instance Directory Structure

```
deploy/openclaws/
  personal-claw/                # Active instance
    config.env                  # Overrides for this instance
    openclaw.json               # Optional: replaces deploy/openclaw.json entirely
  work-claw/                    # Active instance
    config.env                  # Overrides only (uses default deploy/openclaw.json)
  _experimental/                # Disabled (underscore prefix = skip)
    config.env
```

### Config Override Mechanism

Each instance's `config.env` overrides `openclaw-config.env`. No intermediate defaults layer — `openclaw-config.env` IS the defaults.

```
openclaw-config.env                        # All defaults (VPS infra + instance defaults)
  └─ deploy/openclaws/<name>/config.env    # Per-instance overrides

deploy/openclaw.json                       # Default openclaw.json template
  └─ deploy/openclaws/<name>/openclaw.json # Optional: full replacement
```

Shell sourcing: instance `config.env` is sourced after `openclaw-config.env`, overwriting any vars it sets.

### Per-Instance Required Variables

Each instance **must** set (or auto-generate):

- `INSTANCE_GATEWAY_PORT` — unique port (auto-assigned if missing: 18789 + index)
- `INSTANCE_DASHBOARD_PORT` — unique port (auto-assigned if missing: 6090 + index)
- `OPENCLAW_DOMAIN` — unique subdomain for this instance
- `GATEWAY_TOKEN` — unique auth token (auto-generated via `openssl rand -hex 32` if empty)
- `OPENCLAW_INSTANCE_ID` — unique telemetry ID (defaults to dir name)

```yaml

x-openclaw-base: &openclaw-base
  image: openclaw:local
  runtime: sysbox-runc
  read_only: false

    - /tmp:size=1G,mode=1777
    - /var/tmp:size=200M,mode=1777

  user: "0:0"
  security_opt: [no-new-privileges:true]
  entrypoint: ["/app/scripts/entrypoint-gateway.sh"]
  command: ["node","dist/index.js","gateway","--allow-unconfigured","--bind","lan","--port","18789"]
  environment: &base-env
    - NODE_ENV=production
    - TZ=UTC
  logging:
    driver: json-file

    timeout: 10s

    start_period: 300s

services:
  openclaw-personal-claw:
    <<: *openclaw-base
    container_name: openclaw-personal-claw
    deploy:
      resources:
        limits: { cpus: "${PERSONAL_CLAW_CPUS:-3}", memory: "${PERSONAL_CLAW_MEMORY:-5G}", pids: 1024 }
        reservations: { cpus: "2", memory: 2G }
    ports:
      - "127.0.0.1:18789:18789"
      - "127.0.0.1:6090:6090"
    volumes:
      - ./scripts/entrypoint-gateway.sh:/app/scripts/entrypoint-gateway.sh:ro
      - /home/openclaw/sandboxes-home/personal-claw:/home/node/sandboxes-home
      - ./data/personal-claw/docker:/var/lib/docker
      - ./deploy:/app/deploy:ro
      - /home/openclaw/.openclaw/instances/personal-claw:/home/node/.openclaw
    environment:
      - NODE_ENV=production
      - TZ=UTC
      - ANTHROPIC_API_KEY=${PERSONAL_CLAW_AI_GATEWAY_AUTH_TOKEN:-${AI_GATEWAY_AUTH_TOKEN}}
      - ANTHROPIC_BASE_URL=${PERSONAL_CLAW_AI_GATEWAY_WORKER_URL:-${AI_GATEWAY_WORKER_URL}}
      - OPENAI_API_KEY=${PERSONAL_CLAW_AI_GATEWAY_AUTH_TOKEN:-${AI_GATEWAY_AUTH_TOKEN}}
      - OPENAI_BASE_URL=${PERSONAL_CLAW_AI_GATEWAY_WORKER_URL:-${AI_GATEWAY_WORKER_URL}}
      - DASHBOARD_BASE_PATH=${PERSONAL_CLAW_DASHBOARD_BASE_PATH:-}
      - OPENCLAW_DOMAIN_PATH=${PERSONAL_CLAW_OPENCLAW_DOMAIN_PATH:-}
      - TELEGRAM_BOT_TOKEN=${PERSONAL_CLAW_TELEGRAM_BOT_TOKEN:-}

  # openclaw-work-claw: ... (same pattern, different ports + vars)

  # Disable upstream single-instance services
  openclaw-gateway:
    profiles: [single-instance]
  openclaw-cli:
    profiles: [openclaw-cli]

  openclaw-gateway-net:
    external: true
```

The `.env` file at `/home/openclaw/openclaw/.env` gets prefixed vars per instance (e.g., `PERSONAL_CLAW_CPUS=3`), plus shared vars (`AI_GATEWAY_WORKER_URL`).

## Implementation Steps

### Step 1: Create example instance `deploy/openclaws/personal-claw/config.env`

```bash
# personal-claw instance
OPENCLAW_INSTANCE_ID=personal-claw
OPENCLAW_DOMAIN=personal.openclaw.example.com
OPENCLAW_DASHBOARD_DOMAIN=personal.openclaw.example.com
OPENCLAW_DASHBOARD_DOMAIN_PATH=/dashboard
GATEWAY_TOKEN=  # Auto-generated on first deploy
```

### Step 2: Create `deploy/scripts/openclaw-multi.sh`

**Core management script** (~250-350 lines). Commands:

| Command | Description |
|---------|-------------|
| `list` | Show discovered instances (active + disabled) |
| `generate` | Read instance dirs, produce `docker-compose.multi.yml` + update `.env` |
| `deploy-config [name]` | Template openclaw.json + models.json for one or all instances |
| `start` | `generate` + `docker compose up -d` |
| `stop` | `docker compose down` |
| `status` | Show running instance containers |
| `tunnel-config` | Print Cloudflare tunnel rules to configure in Dashboard |

**Key functions:**

- `discover_instances()` — scan `deploy/openclaws/*/`, skip `_`-prefixed, return sorted names
- `load_config(name)` — source layered config: `openclaw-config.env` → `<name>/config.env`
- `assign_ports()` — auto-assign ports for instances that don't specify them (base + alphabetical index)
- `generate_compose()` — emit YAML with anchor + per-instance services
- `generate_env()` — emit prefixed vars into `.env` (e.g., `PERSONAL_CLAW_CPUS=3`)
- `deploy_instance_config(name)` — template openclaw.json + models.json for one instance

### Step 3: Modify `deploy/scripts/setup-infra.sh`

Add multi-instance directory creation after Part 2. When `INSTANCE_NAMES` env var is set:

```bash
# Part 2b: Multi-instance directories
if [ -n "${INSTANCE_NAMES:-}" ]; then
  for name in $INSTANCE_NAMES; do
    sudo -u openclaw bash -c "
      mkdir -p /home/openclaw/openclaw/data/${name}/docker
      mkdir -p /home/openclaw/sandboxes-home/${name}
    "
    sudo mkdir -p /home/openclaw/.openclaw/instances/${name}/{workspace,credentials,logs}
    sudo chown -R 1000:1000 /home/openclaw/.openclaw/instances/${name}
    sudo chown -R 1000:1000 /home/openclaw/sandboxes-home/${name}
    sudo mkdir -p /home/openclaw/.openclaw/instances/${name}/workspace/host-status
    sudo chmod 755 /home/openclaw/.openclaw/instances/${name}/workspace/host-status
  done
fi
```

Backward compatible — when `INSTANCE_NAMES` is empty, single-instance behavior is unchanged.

### Step 4: Modify `deploy/scripts/deploy-config.sh`

Make the config deployment instance-aware. When `INSTANCE_NAME` is set:

- **Config target path**: `/home/openclaw/.openclaw/instances/<name>/` instead of `/home/openclaw/.openclaw/`
- **openclaw.json source**: Check `deploy/openclaws/<name>/openclaw.json` first; fall back to `deploy/openclaw.json`
- **GATEWAY_TOKEN source**: From instance-specific `.env` section or generate new
- **Agent dirs**: Under `instances/<name>/agents/{main,code,skills}/`
- **Shared files (compose, entrypoint, plugins, etc.)**: Only deploy once, skip when `INSTANCE_NAME` is set

Add a new flag: `INSTANCE_NAME=personal-claw deploy-config.sh` runs instance-specific steps only.
Add: `SHARED_ONLY=true deploy-config.sh` runs shared steps only (compose, plugins, scripts, crons).

### Step 5: Modify `deploy/entrypoint-gateway.sh`

Add sandbox image archive loading between dockerd startup and `rebuild-sandboxes.sh` (after line 146):

```bash
# ── 2a. Load pre-built sandbox images from archive ──────────────────
SANDBOX_ARCHIVE="/app/deploy/sandbox-images.tar"
if [ -f "$SANDBOX_ARCHIVE" ] && docker info > /dev/null 2>&1; then
  if ! docker image inspect openclaw-sandbox-toolkit:bookworm-slim > /dev/null 2>&1; then
    echo "[entrypoint] Loading pre-built sandbox images from archive..."
    docker load < "$SANDBOX_ARCHIVE"
    echo "[entrypoint] Sandbox images loaded"
  fi
fi
```

This is a pure optimization — `rebuild-sandboxes.sh` still runs after and verifies/rebuilds if needed. Each instance loads the same archive into its own nested Docker on first boot (~30s vs ~15min build).

### Step 6: Update `deploy/docker-compose.override.yml`

No structural changes. Add a comment at the top noting this file is for single-instance mode only. In multi-instance mode, `docker-compose.multi.yml` is used instead (which incorporates all the same config via YAML anchors).

### Step 7: Update host CLI wrapper

The current `/usr/local/bin/openclaw` wrapper hardcodes `openclaw-gateway`. Replace with an instance-aware wrapper that resolves the target container via priority:

1. `--instance <name>` flag (explicit, stripped before passing to openclaw)
2. Auto-detect: list running `openclaw-*` containers. If exactly one, use it. If multiple, show interactive picker.

```bash
#!/bin/bash
# /usr/local/bin/openclaw — multi-instance CLI wrapper

# 1. Check --instance flag
CONTAINER=""
if [ "$1" = "--instance" ] && [ -n "$2" ]; then
  CONTAINER="openclaw-$2"
  shift 2
fi

# 2. Auto-detect from running containers
if [ -z "$CONTAINER" ]; then
  RUNNING=$(sudo docker ps --filter "name=openclaw-" --filter "status=running" \
    --format '{{.Names}}' | grep -v 'openclaw-cli' | sort)
  COUNT=$(echo "$RUNNING" | grep -c .)

  if [ "$COUNT" -eq 0 ]; then
    echo "No openclaw containers running." >&2; exit 1
  elif [ "$COUNT" -eq 1 ]; then
    CONTAINER="$RUNNING"
  else
    echo "Multiple openclaw instances running:" >&2
    i=1
    while IFS= read -r name; do
      echo "  $i) $name" >&2
      i=$((i + 1))
    done <<< "$RUNNING"
    printf "Select instance [1-%d]: " "$COUNT" >&2
    read -r choice
    CONTAINER=$(echo "$RUNNING" | sed -n "${choice}p")
    if [ -z "$CONTAINER" ]; then
      echo "Invalid selection." >&2; exit 1
    fi
  fi
fi

TTY_FLAG=""
[ -t 0 ] && [ -t 1 ] && TTY_FLAG="-it"
exec sudo docker exec $TTY_FLAG --user node "$CONTAINER" openclaw "$@"
```

In single-instance mode (only `openclaw-gateway` running), this behaves identically to the current wrapper — no flags needed. In multi-instance mode with multiple containers, it prompts for selection.

### Step 8: Update Cloudflare Tunnel docs

Document the per-instance tunnel rules. For each instance, add a public hostname route in CF Dashboard:

| Subdomain | Path | Service | URL |
|-----------|------|---------|-----|
| `personal.openclaw` | `/dashboard/*` | HTTP | `localhost:6090` |
| `personal.openclaw` | catch-all | HTTP | `localhost:18789` |
| `work.openclaw` | `/dashboard/*` | HTTP | `localhost:6091` |
| `work.openclaw` | catch-all | HTTP | `localhost:18790` |

The `tunnel-config` command in `openclaw-multi.sh` prints these rules for easy copy-paste.

## VPS Directory Layout (Multi-Instance)

```
/home/openclaw/
  openclaw/                          # Git repo (shared)
    docker-compose.yml               # Upstream (unused vars cause harmless warnings)
    docker-compose.multi.yml         # GENERATED by openclaw-multi.sh
    .env                             # Shared + per-instance prefixed vars
    data/
      personal-claw/docker/          # Nested Docker storage (instance-specific)
      work-claw/docker/
    deploy/                          # Shared scripts, plugins, toolkit (bind-mounted :ro)
      sandbox-images.tar             # Pre-built archive (optional, speeds up first boot)
    scripts/
      entrypoint-gateway.sh          # Shared entrypoint
  .openclaw/
    instances/
      personal-claw/
        openclaw.json                # Templated config for this instance
        agents/{main,code,skills}/   # Per-agent config + sessions
        workspace/
      work-claw/
        openclaw.json
        agents/{main,code,skills}/
        workspace/
  sandboxes-home/
    personal-claw/code/              # Persistent sandbox home for code agent
    work-claw/code/
```

## Deployment Flow (Multi-Instance)

1. **Local**: Configure `openclaw-config.env` (VPS infra) + create instance dirs in `deploy/openclaws/`
2. **Playbooks 01-03**: Workers, base setup, Docker — unchanged
3. **Playbook 04 § 4.2**: SCP `deploy/` to VPS, run `setup-infra.sh` with `INSTANCE_NAMES="personal-claw work-claw"`
4. **Playbook 04 § 4.3**: Run `deploy-config.sh` with `SHARED_ONLY=true` (shared files once), then per-instance with `INSTANCE_NAME=<name>`
5. **Playbook 04 § 4.4**: `build-openclaw.sh` (builds shared `openclaw:local` image once)
6. **New step**: Run `openclaw-multi.sh generate` to produce compose file + .env vars
7. **Start**: `docker compose -f docker-compose.yml -f docker-compose.multi.yml up -d`
8. **CF Dashboard**: Add tunnel routes per `openclaw-multi.sh tunnel-config` output

## Verification

1. `openclaw-multi.sh list` — shows all discovered instances and their port assignments
2. `openclaw-multi.sh status` — shows running containers, health, resource usage
3. `docker compose -f docker-compose.yml -f docker-compose.multi.yml ps` — all instances healthy
4. `curl -sf http://localhost:<port>/` per instance — gateway responds
5. Each subdomain loads Control UI in browser
6. Device pairing works independently per instance
7. Sandbox commands execute in each instance's isolated nested Docker
