# Fix Multi-Claw Deployment Gaps in Playbooks

## Context

Phases 1-3 established the always-multi-claw architecture: generated `docker-compose.override.yml`, instance layout, dynamic container resolution in scripts/CLI, and updated docs. But the **playbooks** (the instructions Claude follows during deployment) were only updated cosmetically (`openclaw-gateway` → `openclaw-main-claw`). They don't actually support deploying multiple claws.

**Critical bug:** `openclaw-multi.sh generate` is never called during the deploy flow. The `deploy-config.sh` script explicitly skips deploying `docker-compose.override.yml` (line 170: "generated by openclaw-multi.sh, not deployed from staging"), but no playbook ever calls `generate`. For a fresh deploy, `docker compose up -d` will fail because the override file that defines the claw services doesn't exist.

**Scope:** Fix playbooks to work correctly for both single-claw (common case) and multi-claw deployments. Keep the primary flow optimized for single-claw and add multi-claw notes where behavior differs.

---

## Design Decisions

**Playbooks stay single-claw-primary.** The default `main-claw` path should read naturally without multi-claw noise. Multi-claw differences are called out in `> **Multi-claw:**` blockquotes within each section.

**`INSTANCE_NAMES` is auto-discovered.** Playbook 04 discovers claw names from `deploy/openclaws/*/` (excluding `_*` prefixed dirs) rather than requiring manual input. For single-claw (only `main-claw/` exists), this is transparent.

**`openclaw-multi.sh generate` runs ON the VPS** after `deploy-config.sh` syncs the `openclaws/` directory. The script reads from `/tmp/deploy-staging/openclaws/*/` to discover instances.

**Verification uses `docker compose ps`** to discover running containers rather than hardcoding names. Per-claw checks loop over discovered containers.

**Post-deploy handles each claw's token/URL separately** since each claw has its own `GATEWAY_TOKEN` and potentially its own domain.

### 1. Fix `playbooks/04-vps1-openclaw.md` — Deploy Flow

**§4.2 Step 2: Pass INSTANCE_NAMES to setup-infra.sh**

Currently missing from the `env` block (line 155-166). Add:

INSTANCE_NAMES='${INSTANCE_NAMES}'

Before the SSH call, add a step to discover instance names:

INSTANCE_NAMES=$(ls -d deploy/openclaws/*/ 2>/dev/null | xargs -I{} basename {} | grep -v '^_' | tr '\n' ' ')

**§4.2 Step 1: SCP — also copy openclaw-config.env**

`openclaw-multi.sh generate` (called in §4.3 Step 3) needs `openclaw-config.env` at REPO_ROOT. When running from `/tmp/deploy-staging/scripts/`, REPO_ROOT resolves to `/tmp/`. Add to the existing SCP block:

```

**§4.3 Step 3 (NEW): Generate docker-compose.override.yml**

Add between deploy-config.sh (Step 2, line 266) and §4.4 (Build, Start, line 292):

```markdown
### Step 3: Generate Docker Compose override

`deploy-config.sh` deploys per-claw configs but does NOT generate `docker-compose.override.yml`. Run `openclaw-multi.sh generate` to create it from the discovered claws:

\```bash
ssh -i ${SSH_KEY_PATH} -p ${SSH_PORT} ${SSH_USER}@${VPS1_IP} \
  "bash /tmp/deploy-staging/scripts/openclaw-multi.sh generate"
\```

This reads `deploy/openclaws/*/` from staging, assigns ports, and writes `docker-compose.override.yml` + per-instance `.env` variables. The existing shared `.env` (created by `setup-infra.sh`) is preserved.

> **Multi-claw:** If multiple claws exist (e.g., `main-claw/` and `personal-claw/`), each gets its own service in the override file with auto-assigned ports.

> **Cleanup:** `openclaw-config.env` contains secrets (AI_GATEWAY_AUTH_TOKEN). The staging directory cleanup at the end of deploy-config.sh (`rm -rf /tmp/deploy-staging`) handles `deploy/*`, but `/tmp/openclaw-config.env` is separate. Add: `rm -f /tmp/openclaw-config.env` after the generate step.
```

**§4.4 Build, Start, Verify (lines 292-470): Make per-claw aware**

The `docker compose up -d` on line 300 already starts ALL services, which is correct. But the verification commands after it (lines 309, 348, 358, etc.) hardcode `openclaw-main-claw`.

Add a discovery step at the top of §4.4:

# Discover running claw containers
CLAWS=$(sudo docker ps --format '{{.Names}}' --filter 'name=^openclaw-' | grep -v '^openclaw-cli$' | grep -v '^openclaw-sbx-' | sort)

Then replace all hardcoded `openclaw-main-claw` refs in this section with iteration:

```bash
for CLAW in $CLAWS; do
  echo "=== Checking $CLAW ==="
  sudo docker logs --tail 20 "$CLAW"

The sandbox image verification loop (lines 396-456) should test against the FIRST claw only (sandbox images are per-claw but identical). Add a note:

> **Note:** Sandbox images are per-claw (each has its own Docker daemon). This test verifies the first claw; all claws build identical images from the same toolkit config.
```

### 3. Fix `playbooks/08-post-deploy.md` — Per-Claw Post-Deploy

**§8.2 Open the Gateway URL (lines 87-116):**

Currently reads token from shared `.env` and constructs one URL. For single-claw this works fine. For multi-claw, each claw gets its own token baked into its `openclaw.json` by `deploy-config.sh` (lines 82-92: reads from `.env` or generates a new one per claw).

Keep the primary single-claw flow unchanged. Add a multi-claw note:

```
> **Multi-claw:** Each claw has its own gateway token. Read per-claw tokens from the deployed config:
> `sudo python3 -c "import json; print(json.load(open('/home/openclaw/instances/<CLAW_NAME>/.openclaw/openclaw.json'))['gateway']['auth']['token'])"`
> Construct separate URLs for each claw using its domain (from per-claw `config.env`) and token.
```

**§8.3 Device pairing (lines 120-233):** Each claw needs its own pairing. The `openclaw devices list` wrapper auto-detects, so for single-claw it works. For multi-claw:

```
> **Multi-claw:** Repeat device pairing for each claw. Use `openclaw --instance <name> devices list` and `openclaw --instance <name> devices approve <requestId>`.
```

**§8.5 Deployment report (lines 269-421):** Add multi-claw variant for the URLs table:

```
> **Multi-claw:** List each claw's URLs in the table with a "Claw" column.
```

### 4. Fix `playbooks/maintenance.md` — Per-Claw Updates

**Add new section: "Single-Claw Config Update"**

After the Token Rotation section, add:

```markdown
## Updating a Single Claw's Configuration

To update just one claw's `openclaw.json` or `models.json` without affecting other claws:

\```bash
# From local machine
# 1. SCP updated deploy files to VPS
scp -P ${SSH_PORT} -i ${SSH_KEY_PATH} -r deploy/* ${SSH_USER}@${VPS1_IP}:/tmp/deploy-staging/

# 2. Deploy config for one claw only
ssh ... "env INSTANCE_NAME=personal-claw \
  OPENCLAW_DOMAIN_PATH='...' \
  ... \
  bash /tmp/deploy-staging/scripts/deploy-config.sh"

# 3. Restart that claw to pick up new config
sudo -u openclaw bash -c 'cd /home/openclaw/openclaw && docker compose restart openclaw-personal-claw'
\```
```

**Add new section: "Adding a New Claw"**

```markdown
## Adding a New Claw

1. Create `deploy/openclaws/<name>/config.env` (copy from `_example-claw/`)
2. Set per-claw overrides (domain, resources, etc.)
3. SCP deploy files to VPS
4. Run `openclaw-multi.sh generate` to update docker-compose.override.yml
5. Run `deploy-config.sh` for the new claw: `INSTANCE_NAME=<name> bash .../deploy-config.sh`
6. If using CF_API_TOKEN: `openclaw-multi.sh tunnel-config --apply`
7. Start: `docker compose up -d openclaw-<name>`
```
